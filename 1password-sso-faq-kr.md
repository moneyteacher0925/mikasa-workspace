# 1Password 자주 묻는 질문 (FAQ)

## SSO로 1Password 잠금 해제 — 관리자용

---

### SSO로 1Password 잠금 해제는 얼마나 안전한가요?

SSO 잠금 해제는 1Password의 제로 지식(Zero-Knowledge) 아키텍처와 엔드투엔드 암호화를 유지하며, 복호화는 여전히 기기에서 이루어집니다. 다만, 위험 평가를 수행할 때 표준 1Password 모델과 보안 모델 및 위험 요소가 어떻게 다른지 이해하는 것이 중요합니다.

ID 제공자(IdP)를 통한 잠금 해제는 기존 1Password 보안 모델 위에 추가적인 본인 확인 계층을 제공합니다. 1Password는 직원의 암호화된 자격 증명과 기기 키를 사용하여 SSO로 잠금 해제하므로, 등록 과정이 간소화되고 계정 비밀번호가 필요 없어집니다.

모델 및 관련 위험에 대한 자세한 내용은 "SSO 잠금 해제 보안 모델" 지원 문서를 참조하세요.

---

### 1Password의 SSO 접근 방식은 경쟁사와 어떻게 다른가요?

대부분의 기업용 비밀번호 관리자는 SSO를 지원할 때 두 가지 접근 방식 중 하나를 사용합니다.

첫 번째는 인증 브릿지(Auth Bridge)로, 공격자에게 크고 매력적인 목표가 되며 고객이 온프레미스 인프라를 유지해야 합니다. 두 번째는 모든 사용자 데이터에 대한 공유 암호화 키로, 한 명의 직원이 침해되면 회사 전체가 위험에 처합니다. 두 접근 방식 모두 1Password의 엄격한 보안 요구 사항을 충족하지 못합니다.

1Password는 신뢰할 수 있는 기기 모델(Trusted Device Model)을 사용합니다. 즉, ID 제공자 자격 증명이 유출되더라도 공격자는 1Password 데이터에 접근할 수 없습니다. SSO 잠금 해제는 비밀 키(Secret Key)의 필요성을 제거하되, 모든 데이터를 기기에서 안전하게 유지하는 방식으로 수행합니다. 악의적인 행위자가 데이터에 접근하려면 여전히 신뢰할 수 있는 기기가 필요합니다. 데이터는 보호된 상태로 유지되며, 새 기기에 로그인하는 것이 더 쉬워집니다.

이 접근 방식은 제로 지식을 유지하며 엔드투엔드 암호화됩니다. 복호화는 여전히 기기에서 이루어집니다. **1Password는 데이터를 복호화하는 데 필요한 키를 저장하거나 접근할 수 없습니다.**

---

### SSO 잠금 해제로 전환한 사용자는 기존에 1Password에 저장한 ID 제공자 비밀번호를 어디에 보관해야 하나요?

SSO 잠금 해제를 활성화하면, 1Password 로그인이 ID 제공자 로그인으로 대체됩니다. ID 제공자에는 1Password의 비상 키트(Emergency Kit)같은 것이 없지만, 비상 키트 보관 방법과 유사한 방식으로 ID 제공자 비밀번호를 저장할 수 있습니다.

ID 제공자용 비밀번호를 무작위이면서도 기억하기 쉬운 것으로 만드는 것을 권장하며, 1Password가 이를 도와줄 수 있습니다. 이렇게 하면 비밀번호를 기억 외의 다른 곳에 저장할 필요가 없어집니다. 그 비밀번호가 기억해야 할 유일한 비밀번호가 됩니다.

---

### SSO 잠금 해제는 자동 프로비저닝도 하나요, 아니면 SCIM 브릿지가 여전히 필요한가요?

자동 사용자 프로비저닝을 위해서는 여전히 1Password SCIM 브릿지가 필요하며, 환경 내에 브릿지를 배포해야 합니다.

SSO 잠금 해제는 별도의 구성이 필요하며, SCIM 브릿지가 제공하는 자동 사용자 프로비저닝과 직접적으로 연결되지 않습니다. SSO로 1Password를 잠금 해제하기 위해 별도의 인프라를 설정할 필요는 없습니다.

이 기능들은 별도로 구성되며, SCIM 브릿지 없이 SSO만으로 1Password를 사용할 수도 있지만, 최고의 보안과 자동화 경험을 위해 두 가지를 함께 사용하는 것을 권장합니다.

---

### 기존 사용자가 계정 비밀번호에서 SSO 잠금 해제로 (또는 그 반대로) 어떻게 전환하나요?

계정 비밀번호에서 SSO 잠금 해제로의 전환 과정은 간단합니다. 사용자가 SSO로 잠금 해제하도록 설정되면, 다음 로그인 시 팝업이 표시됩니다.

그런 다음 ID 제공자에 로그인하고 전환을 완료하는 안내를 따르기만 하면 됩니다. 전환 과정에서 사용자의 기기가 첫 번째 신뢰할 수 있는 기기가 됩니다. 다음에 로그아웃했다가 다시 로그인하면, 계정 비밀번호 대신 SSO로 잠금 해제하는 옵션이 표시됩니다. 이 시점부터 더 이상 계정 비밀번호, 비밀 키, 비상 키트를 관리할 필요가 없습니다.

SSO를 비활성화하고 계정 비밀번호와 비밀 키를 다시 설정하려면 계정 복구 과정을 거쳐야 합니다. 사용자는 계정 복구를 완료하고, 새 계정 비밀번호를 설정하고, 새 비밀 키와 비상 키트를 받기 위한 이메일을 받게 됩니다. 참고: 계정 소유자(Owner)는 SSO로 잠금 해제할 수 없습니다. 이는 실수로 인한 잠금을 방지하기 위한 것입니다.

---

### SSO 신뢰할 수 있는 기기 워크플로우와 인증 코드를 비활성화할 수 있나요?

아니요, SSO 신뢰할 수 있는 기기 워크플로우는 비활성화할 수 없습니다. 신뢰할 수 있는 기기 등록 과정은 1Password의 엔드투엔드 암호화의 핵심입니다.

**기기의 정의:**
- 웹 브라우저 (Safari, Firefox, Chrome, Brave, Arc 등)
- 1Password 8 데스크톱 앱 (Windows, Linux, macOS)
- 1Password 8 for Android
- 1Password 8 for iOS

**항상 두 개의 기기에 로그인할 것을 권장합니다. 이상적으로는:**
1. 1Password 계정을 생성한 브라우저
2. Windows, Mac 또는 Linux용 1Password

SSO 잠금 해제의 보안 모델과 신뢰할 수 있는 기기 프로세스에 대한 자세한 내용은 1Password 블로그를 참조하세요. 신뢰할 수 있는 기기 설정 방법은 1Password 지원 페이지에서도 확인할 수 있습니다.

---

### 왜 1Password는 ID 제공자나 MDM 도구의 신뢰할 수 있는 기기 데이터를 사용할 수 없나요?

이 접근 방식에는 보안/개인정보 위험이 있습니다. ID 제공자나 MDM 도구에 연결하면 고객의 데이터셋에 대한 가시성이 생기며, 이는 1Password 측에 상당한 책임을 부과하고 핵심 개인정보 보호 가치를 위반하게 됩니다.

이 기능을 제거하면 엔드투엔드 암호화가 약화되어 사이버 공격에 취약해집니다. 기기 승인 과정이 없으면 MITM(중간자) 공격이나 악의적인 1Password 직원이 요청된 키를 교체하여 사용자 데이터에 완전한 접근 권한을 얻을 수 있습니다.

---

### 두 개의 다른 ID 제공자를 SSO에 사용할 수 있나요?

아니요, SSO 제공자로 하나의 ID 제공자를 선택해야 합니다. SSO 인증에 여러 ID 제공자를 사용할 수 없습니다. 최상의 경험을 위해 SCIM 브릿지를 통한 계정 프로비저닝에 사용하는 것과 동일한 ID 제공자를 사용하는 것을 권장합니다.

---

### 일부 직원에게만 SSO 잠금 해제를 활성화할 수 있나요?

네. 1Password 계정 내에서 SSO로 잠금 해제할 수 있는 그룹을 지정할 수 있습니다. 사용자 정의 그룹과 기본 내장 그룹(예: 관리자, 팀 멤버)을 포함하며, 계정 소유자(Owner)는 제외됩니다. 게스트 사용자를 SSO에서 제외하는 옵션도 있습니다.

소유자(Owner)는 SSO로 잠금 해제할 수 없으며 계정 비밀번호와 비밀 키를 사용해야 합니다. 이는 소유자가 접근 권한을 잃고 다른 사람이 복구할 수 없는 경우의 실수로 인한 잠금 및 데이터 손실을 방지하기 위한 것입니다.

---

### ID 제공자가 다운되면 직원들이 여전히 1Password에 접근할 수 있나요?

SSO 잠금 해제가 설정되고 생체 인식이 활성화된 사용자는 오프라인이거나 ID 제공자가 다운된 경우에도 1Password 클라이언트 앱(데스크톱 및 모바일)에 접근할 수 있습니다. ID 제공자 장애 시 계정 소유자는 여전히 비밀번호와 비밀 키로 로그인할 수 있습니다. 긴급 상황에서 소유자는 SSO 정책을 조정하여 사용자가 비밀번호와 비밀 키 인증을 다시 설정하도록 할 수 있습니다.

---

### SSO 잠금 해제가 활성화된 사용자에게 여전히 2FA를 강제할 수 있나요?

SSO 잠금 해제에 등록된 사용자에게는 1Password 계정에서 더 이상 2FA가 강제되지 않습니다.

대신, ID 제공자가 사용자의 2FA 요구 사항을 관리하게 됩니다. ID 제공자에서 설정한 2FA 관련 강제 및 설정이 SSO로 1Password를 잠금 해제하는 사용자에게 적용됩니다.

---

### SSO 잠금 해제가 활성화된 상태에서 1Password에 방화벽 규칙을 설정할 수 있나요?

방화벽 규칙은 사용자가 SSO로 잠금 해제하도록 설정되어 있더라도, 1Password 비즈니스 계정의 보안 섹션에서 계속 적용할 수 있습니다.

ID 제공자에서도 방화벽 규칙을 설정할 수 있으며, 이 규칙은 사용자가 SSO로 1Password를 잠금 해제할 때(즉, ID 제공자에 인증할 때) 적용됩니다. 1Password 잠금 해제 후에는 1Password에 설정된 방화벽 규칙이 적용됩니다.

---

### SSO 잠금 해제의 유예 기간은 어떻게 계산되나요?

선택된 그룹의 경우, 유예 기간은 SSO 잠금 해제 구성 설정에 그룹을 추가할 때 시작됩니다. 유예 기간이 만료된 후에도 사용자를 그룹에 추가할 수 있다는 점에 유의하세요. 이 경우 사용자는 복구 상태에 놓이며 SSO를 구성하기 위해 복구 과정을 완료해야 합니다.

그룹의 유예 기간이 만료되었는데 계속 전환 유예 기간을 제공하고 싶다면, 새 그룹을 만들어 SSO 정책에 추가해야 합니다.

"전체(Everyone)" 옵션을 선택한 경우, 유예 기간은 선택한 날부터 시작됩니다.

---

### 왜 ID 제공자가 아닌 1Password에서 계정 복구를 수행해야 하나요?

ID 제공자에서의 계정 복구는 사용자가 ID 제공자 자격 증명을 기억하지 못할 때 발생합니다. 반면, 사용자가 1Password의 신뢰할 수 있는 기기에 더 이상 접근할 수 없는 경우에는 1Password에서 복구를 수행합니다. 이는 주로 신뢰할 수 있는 기기 보안 모델의 결과입니다.

1Password에서 계정 복구가 필요한 가장 일반적인 시나리오는 사용자가 신뢰할 수 있는 기기를 하나만 설정한 경우입니다. 해당 기기에 대한 접근을 잃으면, 1Password 관리자에게 연락하여 계정 복구를 시작해야 합니다. 이를 방지하려면 모든 사용자에게 1Password 데스크톱 앱을 배포하고 해당 앱을 새로운 신뢰할 수 있는 기기로 등록하는 것을 권장합니다. 두 번째 신뢰할 수 있는 기기가 설정되어 있으면, 사용자는 신뢰할 수 있는 기기 인증 코드를 검색하여 비인가된 추가 기기를 다시 등록할 수 있습니다.

### 사용자들이 매일 새 컴퓨터나 가상 Windows 환경에 로그인합니다. 매번 SSO로 로그인하는 것을 방지하려면 어떻게 하나요?

데이터 지속성이 없는 환경은 1Password에 어려운 환경입니다. 1Password 소프트웨어는 신뢰할 수 있는 기기에 데이터를 저장하고 처리하는 것에 의존하기 때문입니다. 이는 보안 모델의 균형의 일부입니다. 이런 환경에서 1Password를 효과적으로 실행할 수 있을 수도 있고 없을 수도 있지만, Windows에서 시도해볼 수 있는 몇 가지 방법이 있습니다. 때때로 이런 컴퓨터를 운영하는 회사에서는 특정 데이터를 동기화하여 사용자가 로그인할 때마다 사용할 수 있도록 할 수 있습니다.

**브라우저 확장 프로그램의 경우,** 브라우저에 확장 프로그램이 자동으로 설치되도록 하고 다음 디렉토리를 동기화하여 사용자 계정이 표시되도록 할 수 있습니다:

- Chrome 경로: *%LocalAppData%\Google\Chrome\UserData\Default\IndexedDB*
  (Edge도 동일하며, `Google\Chrome`을 `Microsoft\Edge`로 변경)

**Windows 데스크톱 앱의 경우:**
- .msi 설치 프로그램으로 설치한 경우: *%ProgramFiles%* 에 저장
- .exe 설치 프로그램으로 설치한 경우: *%LocalAppData%* 에 저장
