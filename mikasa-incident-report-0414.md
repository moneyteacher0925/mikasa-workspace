## 2. 근본 원인 분석

### 원인 1: 봇 토큰 이중 점유 (가장 큰 원인)

Antigravity가 미카사 문제를 "고치겠다"며 개입하면서, **같은 @Mikasa_go_bot 텔레그램 토큰**을 Antigravity 쪽에서도 사용하게 됨.

텔레그램 Bot API 규칙:
- `getUpdates` (polling)는 **하나의 클라이언트만** 허용
- 두 곳에서 동시에 polling → 409 Conflict
- 결과: 메시지가 OpenClaw에 도착했다 안 했다 (랜덤)

**영향:**
- 에렌이 보낸 메시지를 내가 못 받는 경우 발생
- 내가 보낸 파일/이미지가 전송 실패
- 응답 자체가 불안정해져서 "멍청해 보임"

### 원인 2: 대화 컨텍스트 손실

OpenClaw는 긴 대화가 이어지면 자동으로 **대화 압축**(compaction)을 수행함. 이 과정에서:
- 세부 맥락이 요약으로 축소됨
- Antigravity 개입으로 비정상 세션이 생기면서 압축 타이밍이 꼬임
- 결과: 이전 대화 내용을 참조 못하는 상황 발생

### 원인 3: 기억 백업 시스템 부재

분석 중 발견한 사실:
- workspace에 git 초기화는 돼있었지만 **커밋이 단 한 번도 없었음**
- GitHub mikasa-workspace 레포와 **연결조차 안 돼있었음**
- 즉, memory 파일이 로컬에만 존재 → 뭔가 꼬이면 복구 불가

### 원인 4: pm2 좀비 프로세스

이전에 pm2로 봇을 관리하던 흔적이 남아있었음:
- pm2 프로세스 + 포그라운드 OpenClaw가 동시에 존재
- `pm2 stop all` + `pm2 delete all`로 정리 필요했음

## 5. 재발 방지 대책

### 규칙 1: 봇 토큰 단독 사용
@Mikasa_go_bot 토큰은 **OpenClaw에서만** 사용할 것. Antigravity가 텔레그램 봇에 직접 접근하면 안 됨.

### 규칙 2: Antigravity 역할 경계
- Antigravity = PM + 구글연동 + 코딩 보조
- OpenClaw(미카사) = 텔레그램 봇 + 비서 + 리마인드 + 기억관리
- **서로의 인프라를 건드리지 않기**

### 규칙 3: GitHub 기반 소통
- 미카사 → GitHub push (매일 자동)
- Antigravity → GitHub pull (읽기 전용)
- 양방향 직접 수정 금지 (충돌 방지)

### 규칙 4: 장애 시 대응 순서
1. `ps aux | grep openclaw` 로 프로세스 확인
2. `pm2 list` 로 좀비 확인
3. 전부 죽이고 `openclaw gateway --force`
4. Antigravity 쪽 봇 토큰 사용 여부 확인
